# Internationalization

## What it is / why it exists

Two locales are supported: **`en`** (default) and **`tr`**. URLs always carry the locale prefix (`/en/...`, `/tr/...`). New visitors are redirected to their best-fit locale on first request; user choice is then persisted in a cookie.

Implementation is built on [next-intl](https://next-intl.dev) with a thin custom locale resolver in front.

## Source of truth

- Routing config: [i18n/routing.ts](../../i18n/routing.ts)
- Server-side message loader: [i18n/request.ts](../../i18n/request.ts)
- Typed navigation helpers: [i18n/navigation.ts](../../i18n/navigation.ts)
- Plugin wiring: [next.config.ts](../../next.config.ts)
- Resolver + redirect: [proxy.ts](../../proxy.ts)
- Catalogs: [messages/en.json](../../messages/en.json), [messages/tr.json](../../messages/tr.json)

## Locale resolution order

[proxy.ts](../../proxy.ts) decides the locale for any request that does **not** already have a `/en` or `/tr` prefix:

1. `NEXT_LOCALE` cookie (if its value is a known locale)
2. `x-vercel-ip-country` or `cf-ipcountry` header equals `TR` → `tr`
3. `Accept-Language` header contains `tr` or `tr-TR` → `tr`
4. Fallback → `en`

The matched locale is set as the cookie on the redirect response (1-year expiry, `SameSite=Lax`). If the request **already has** a locale prefix, the proxy delegates to next-intl's middleware unchanged.

## Rules and invariants

- **Same key tree in both catalogs.** A key that exists in `en.json` must exist in `tr.json` (and vice versa). Missing keys break TypeScript inference and produce runtime errors.
- **Server pages must call `setRequestLocale(locale)`** at the top after reading `params`. Without it, `generateStaticParams` cannot prerender that route.
- **Never import `next/link` for in-app navigation.** Use the wrapped `Link` from [i18n/navigation.ts](../../i18n/navigation.ts) so locale prefixes are preserved.
- **Locale type is closed.** Use `AppLocale` (and `isAppLocale()` for untrusted strings) from [i18n/routing.ts](../../i18n/routing.ts) — don't widen to `string`.

## Common patterns

### Adding a string

1. Decide the namespace (e.g. `landing.subtitle`).
2. Add the key to [messages/en.json](../../messages/en.json) and [messages/tr.json](../../messages/tr.json) at the same path.
3. In a server file:

```ts
const t = await getTranslations("landing");
return <p>{t("subtitle")}</p>;
```

In a client file:

```ts
"use client";
const t = useTranslations("landing");
return <p>{t("subtitle")}</p>;
```

### Switching locale on the client

```ts
"use client";
import { useRouter, usePathname } from "@/i18n/navigation";

const router = useRouter();
const pathname = usePathname();
router.replace(pathname, { locale: "tr" });
```

Wrap in `useTransition` so the toggle button stays responsive — see [components/layout/language-switcher.tsx](../../components/layout/language-switcher.tsx).

### Adding a new locale

1. Add the locale code to `routing.locales` in [i18n/routing.ts](../../i18n/routing.ts).
2. Create `messages/<locale>.json` with the same keys.
3. Mirror per-locale content files in [content/](../../content/) (e.g. `profile.<locale>.json`, `projects.<locale>.json`) and add a branch in [lib/data.ts](../../lib/data.ts).
4. Add a country / `Accept-Language` rule to `detectLocale` in [proxy.ts](../../proxy.ts) if a regional default applies.

## Gotchas

- **`localeDetection` is `false` in `routing`.** Detection happens in our proxy, not in next-intl middleware. Re-enabling it would double-handle the redirect.
- **Cookie name is `NEXT_LOCALE`.** This is what next-intl reads if locale detection ever moves back to the library.
- **`localePrefix: "always"`** means `/` is never a valid landing URL — the proxy redirects it. Anything that hard-codes `/` for an in-app link must use `Link` from `@/i18n/navigation` to get the prefix.
- **Server-only catalog imports.** [i18n/request.ts](../../i18n/request.ts) dynamically imports the catalog. Don't switch to a static import or both files end up in the client bundle.
